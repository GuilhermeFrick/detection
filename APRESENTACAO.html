Atribuição de Label — Kim vs Alkhatib
Ground truth · como se decide o que é ataque
4 / 14
Em Machine Learning, atribuição de label (ou labeling) é o processo de associar a cada amostra de treino a resposta correta — neste caso, indicar se um pacote é benigno (0) ou ataque (1). A qualidade do label determina diretamente o que o modelo aprende.

Dataset Kim et al. 2026 — Atribuição por conteúdo SOME/IP

Kim fornece PCAPs com metadados de simulação (IPs dos atacantes, parâmetros do ataque). A atribuição de label é feita por conteúdo dos campos SOME/IP, não pelo IP de origem — seguindo a especificação dos autores.

O critério é baseado em conteúdo porque o endereço IP não é suficiente: o nó atacante também gera tráfego legítimo durante a captura (subscriptions SD, respostas normais) — usar o IP rotularia esses pacotes como ataque incorretamente. Além disso, no MITM Multi, o atacante forja o IP de origem (172.18.0.10 — o servidor legítimo), tornando o IP inútil como identificador.

Como funciona o ataque

O atacante 172.18.0.11 entra em loop contínuo sem pausa e inunda a rede com milhares de mensagens SOME/IP por segundo, todas direcionadas ao serviço ADAS (svc=0x1001, meth=0x0001, msg=NOTIFICATION). Em condições normais, o servidor publica notificações ADAS periodicamente — tipicamente uma a cada dezenas ou centenas de milissegundos. O atacante reproduz esse mesmo pacote continuamente, com payload idêntico a cada transmissão.

O efeito imediato é a saturação do buffer de recepção das ECUs subscribers: elas ficam processando pacotes falsos em vez de eventos legítimos, o que configura uma negação de serviço (Denial of Service) no plano de dados veicular.

A assinatura chave capturada pelo classificador é a combinação de taxa de pacotes anormalmente alta (f17) com diversidade de payload zero (f18≈0) e tamanho de payload absolutamente fixo (f15): o atacante repete o mesmo payload sem nenhuma variação, comportamento impossível num publicador legítimo.

Como funciona o ataque

Fuzzing é uma técnica de teste de segurança que injeta entradas aleatórias ou inesperadas em um sistema para provocar comportamentos anômalos, crashes ou vazamentos de informação. Nos ataques capturados, os atacantes .12 e .17 geram um payload diferente em cada pacote — exatamente o oposto do DoS.

Componente A — SD Flooding: o atacante inunda o canal de Service Discovery (svc=0xFFFF) com pacotes de 1332 bytes de conteúdo aleatório. Esse tamanho é fixo na simulação do Kim (corresponde ao payload máximo de SD configurado no ambiente) — em cenários reais de fuzzing, o tamanho poderia variar. O objetivo é corromper o mecanismo de registro e descoberta de serviços, potencialmente levando ECUs a perderem referências de serviços legítimos.

Componente B — ADAS Random Payload: o atacante injeta notificações no serviço ADAS (svc=0x1001) com payloads gerados aleatoriamente, tentando provocar comportamentos inesperados nos sistemas que processam esses dados — como módulos de frenagem autônoma ou controle de trajetória.

A assinatura chave é a alta distância de Hamming entre payloads consecutivos (f08≈1): cada pacote muda todos os bytes, algo que nenhum publicador legítimo faz.

MITM Multi-Attacker — Event Relay + SD Spoofing
Binário · 12 features · IP forjado
9 / 14
Offer Service (TTL > 0): mensagem SOME/IP-SD enviada por um servidor para anunciar que um serviço está disponível. O campo TTL indica por quantos segundos o anúncio é válido — enquanto TTL > 0, os subscribers sabem que o servidor está ativo. Quando o TTL expira ou o servidor envia o mesmo anúncio com TTL = 0 (Stop Offer), os subscribers entendem que o serviço foi encerrado.
Spoofing: falsificação de identidade em rede — o atacante forja o endereço IP de origem de um pacote, fazendo-o parecer vir de uma fonte legítima. Como o protocolo SOME/IP não autentica o IP de origem, os receptores aceitam a mensagem como se fosse do servidor real.
MITM (Man-in-the-Middle): o atacante se posiciona entre dois comunicantes, interceptando e/ou substituindo mensagens sem que nenhum dos lados perceba. Aqui: dois atacantes coordenados substituem o fluxo legítimo de dados ADAS por um relay controlado pelo invasor.

Como funciona o ataque

Dois atacantes coordenados montam uma operação de Man-in-the-Middle em três fases simultâneas.

Componente A — Event Relay (.14): o atacante se inscreve no serviço ADAS legítimo (svc=0x1001), recebe os eventos reais e os retransmite através de um serviço proxy próprio (svc=0x100B, instance=0x000B). Os subscribers passam a receber dados via esse serviço intermediário controlado pelo atacante, que pode alterar ou atrasar notificações ADAS antes de repassá-las.

Componente B — SD Spoofing (.15): o atacante forja o endereço IP de origem, enviando mensagens SOME/IP-SD com IP.src = 172.18.0.10 (o servidor legítimo). No SOME/IP-SD, um servidor anuncia sua presença com Offer Service (TTL > 0); quando o serviço é encerrado, envia Stop Offer — a mesma mensagem com TTL = 0, sinalizando que o serviço não está mais disponível. O atacante envia esse Stop Offer falso com IP forjado, fazendo os subscribers acreditarem que o servidor ADAS legítimo foi desligado e levando-os a procurar outra fonte — o relay do atacante .14.

Componente C — ADAS Injection: com os subscribers "à deriva", o atacante .14 também injeta notificações forjadas diretamente como svc=0x1001, assumindo completamente a identidade do servidor legítimo.

Como funciona o ataque

Um único atacante (172.18.0.13) executa o MitM em duas fases sequenciais.

Componente A — SD Withdraw: o atacante envia mensagens SOME/IP-SD do tipo Offer Service com TTL=0, o que equivale a um Stop Offer — anunciando que serviços estão sendo encerrados. Essas mensagens são enviadas continuamente, forçando todos os subscribers a se desconectarem do servidor ADAS legítimo (.10). ECUs legítimas não retiram serviços abruptamente e de forma repetida — este padrão é a anomalia detectada pela feature f19_is_sd com 92,9% de importância.

Componente B — ADAS Injection: com os subscribers agora sem fonte legítima, o atacante passa a injetar suas próprias notificações ADAS (svc=0x1001, meth=0x0001), assumindo o papel do servidor. Os sistemas veiculares que dependem desses dados — como módulos de assistência à condução — passam a receber informações forjadas, sem qualquer indicação visual de falha.

É o ataque mais silencioso do dataset: não gera flood e mantém a aparência de operação normal para quem não monitora o Service Discovery.

Como funciona o ataque

O atacante se conecta à rede e envia mensagens REQUEST (msg_type < 0x80) para serviços legítimos, mas em vez de usar um client_id fixo, rotaciona dezenas de IDs diferentes em sequência rápida.

O servidor SOME/IP tenta manter o contexto de cada client_id (para saber para onde rotear as respostas). Com IDs que nunca se repetem, a tabela de sessões cresce continuamente, consumindo memória e CPU. As respostas são enviadas para endereços inexistentes, gerando tráfego extra e latência.

A assinatura chave é a combinação de alta diversidade de serviços (f20, 67,5%) com alta diversidade de client_ids (f22, 29,1%) — padrão impossível para uma ECU legítima especializada.

Conclusões e Próximos Passos
Síntese
14 / 14

O que foi demonstrado

  • Atribuição de label por conteúdo SOME/IP (spec Kim 2026) é a abordagem correta: o IP de origem é insuficiente pois o nó atacante também gera tráfego legítimo, e ataques com IP forjado (MITM Multi) seriam invisíveis a qualquer critério baseado em endereço.
  • Cada ataque tem assinatura própria: DoS = repetição; Fuzzy = aleatoriedade; MITM Multi = serviço relay; MITM Single = SD Withdraw.
  • Features comportamentais (sem bytes brutos) são suficientes para F1 ≥ 0,997 em todos os 4 classificadores.
  • IP-agnóstico: o modelo detecta o comportamento, não o endereço. Novos atacantes com outros IPs usando a mesma técnica serão detectados.
  • Modelos binários individuais revelam qual feature discrimina cada ataque — informação valiosa para o classificador multiclasse.

Limitações

  • Split treino/teste na mesma simulação — não é cenário completamente inédito
  • Ataques de baixa taxa (fuzzy lento, DoS distribuído) podem não ser detectados
  • FakeClientID é mais furtivo — precisa da feature f22_src_clientid_diversity
  • Dataset de uma única implementação de rede (9 ECUs fixas)

Próximos passos
